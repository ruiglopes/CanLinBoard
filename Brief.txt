This should build to be flashed with the bootloader project in https://github.com/ruiglopes/2350Bootloader.git

It should use can2040 as CAN bus implementation https://github.com/KevinOConnor/can2040

It shouldn't use blocking code such as sleep or wait commands.

All of its features should be configurable and stored in non-volatile memory

An external windows software should be created to interface with the hardware and change the different parameters.

From a CAN point of view, CAN 1 should always be available, CAN 2 should be switchable. Baud rate and termination resistors should be configurable for all buses.

From a LIN point of view, all the buses will also require configuration capability. They should be switchable between master and slave and the baud rate should be configurable. A scheduling table should be available if set up as master.

Some of the high level features of the board:

Act as a gateway between CAN and LIN, between both CAN buses or between both LIN buses. Initially pass through complete frames, but ultimately it should have the ability of passing specific bytes or even modify them "in transit"

We would aim to implement base features such as native support for the Bosch Motorsport WDA LIN wiper motor or the pierburg CWA400 LIN coolant pump.

All the buses, CAN and LIN, should have diagnostics and watchdogs to monitor their health

A configurable CAN message with diagnostics and statuses will also be required.

An example of a product that uses a similar principle of firmware https://github.com/corygrant/DingoPDM_FW configuration being set with an external tool https://github.com/corygrant/dingoConfig for your reference
